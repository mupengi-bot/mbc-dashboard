World monitor 웹사이트 분석해서 지방선거 모니터링 사이트 만드는 방법 찾아줘

동네소년, “World Monitor” 스타일(= 수십~수백 개 소스를 한 화면에 ‘상황판’처럼 통합하고, 지도/레이어/실시간 피드/요약/알림으로 과부하를 줄이는 방식)을 그대로 빌려서 지방선거 모니터링 사이트로 만드는 방법을, 분석 → 설계 → 구축 절차로 정리해볼게요.

1) World Monitor에서 “지방선거 모니터링”에 바로 가져올 핵심 설계 요소
World Monitor README가 말하는 가치(문제→해결)는, 선거 모니터링에도 거의 그대로 적용됩니다.
(1) “흩어진 정보 → 단일 대시보드”
World Monitor는 “100+ 소스의 뉴스가 흩어져 있어 파편화” 문제를 단일 대시보드로 해결한다고 명시합니다. (GitHub)
➡️ 지방선거에선 “언론 기사 + 후보 SNS + 여론조사 + 선관위 공지 + 지역 이슈(사건/재난/교통) + 팩트체크”가 흩어져 있는 게 똑같은 문제예요.
(2) “지리(Geo) 컨텍스트 + 레이어”
World Monitor는 인터랙티브 맵 + 다수(40+) 레이어를 핵심 가치로 둡니다. (GitHub)
➡️ 지방선거는 행정구역(시·군·구/선거구) 기반이므로 “지도 중심 UI”가 특히 잘 맞습니다.
(3) “정보 과부하 → AI 요약(brief) + 포커스 포인트”
World Monitor는 AI-synthesized briefs(요약 브리프) + focal point detection으로 과부하를 줄인다고 설명합니다. (GitHub)
➡️ 선거 모니터링에서도 “오늘 뜬 지역 핵심 의제 TOP N”, “후보별 리스크/논쟁 포인트”, “급상승 키워드” 같은 브리핑이 메인 가치가 됩니다.
(4) “로컬 LLM + 로컬 인덱스(RAG)”
“AI를 로컬에서 돌릴 수 있고(예: Ollama/LM Studio), Headline Memory가 로컬 semantic index를 만들어 RAG 쿼리를 한다”는 설명이 있습니다. (GitHub)
➡️ 선거 데이터는 민감도가 높을 수 있어서 외부로 안 나가게(온프레/로컬) 설계하는 니즈가 큽니다.
(5) “API를 먼저 설계(Proto-first)”
“Proto-first API contracts… typed services + OpenAPI”로 깨지기 쉬운 스크래핑 대신 계약 기반 API를 강조합니다. (GitHub)
➡️ 지방선거 모니터링도 “수집→정제→분석→대시보드”가 분업되기 쉬우니, 데이터 계약을 먼저 잡는 게 유지보수 핵심이에요.

2) 지방선거 모니터링 사이트 기능을 “World Monitor식”으로 재구성
아래는 MVP(최소기능) 기준으로, World Monitor의 구성 철학을 선거판에 맞게 옮긴 기능 블록입니다.
A. 메인 화면(상황판)
Breaking / Hot(급상승)
지역별 급증 키워드(예: “○○역 개발”, “의혹”, “공약 철회”)
급증의 근거: 기사량/SNS 언급/댓글량/조회수 변화율
The Map(지도)
선거구 레이어(시·군·구/읍면동/국회의원 지역구 등 선택)
이슈 레이어: 사건사고/민원 다발/재난/교통/주요 공약(개발·복지·교육)
후보 레이어: 유세 일정/공약 위치태그/언론 노출 핫스팟
The Wall(피드 월)
매체 피드(언론사/지역 커뮤니티/유튜브/공식 보도자료)
후보 피드(공식 계정 포스트/영상 업로드/라이브)
“필터”: 지역/후보/정책 분야/논쟁(리스크)/팩트체크
Brief(오늘의 브리핑)
“오늘 충북 TOP 5 이슈”
“후보별 긍·부정 논쟁 포인트 3개”
“허위정보 의심 신호(확정 아님)” 리스트
B. 알림(Alerts)
룰 기반: “후보 A + 키워드 ‘의혹’ 포함 기사 10분 내 5건 이상”
추세 기반: “평소 대비 3σ 이상 급증”
채널: 이메일/슬랙/카톡(가능하면 사내 메신저)/웹푸시
C. 아카이브/검색
“후보·이슈 타임라인”
“발언/공약 원문 + 출처 링크 + 유사 발언”
벡터 검색(RAG): “○○ 개발 공약 관련 논쟁 정리해줘(출처 포함)”

3) 데이터 파이프라인(수집→정제→분석) 설계: ‘깨지지 않게’ 만드는 법
선거 모니터링은 수집이 제일 먼저 망가집니다. 그래서 World Monitor의 “프로토-퍼스트/깨지기 쉬운 API 지양” 철학을 참고해, 아래 순서가 안전합니다. (GitHub)
1) 수집(Ingestion)
권장 소스 우선순위
공식 RSS/공식 API (언론사 RSS, 공공데이터/선관위 공지 등)
플랫폼 제공 공유 기능/임베드 (유튜브 채널 RSS/공식 공유 링크 기반)
크롤링은 최후(약관/로봇/차단/법적 이슈)
특히 SNS(인스타/페북/X)는 약관·접근제한이 잦으니 “공식 API/합법적 수집” 범위를 먼저 확정하는 게 중요합니다.
2) 정규화(Normalization)
공통 스키마 예시
source_type(news/sns/press/notice)
region_code(선거구 코드)
actor(후보/정당/기관)
topic(경제/복지/개발/교육/안전 등)
text, url, published_at
engagement(가능하면)
여기까지가 “데이터 계약(API contract)” 영역입니다. (이게 없으면 대시보드가 흔들려요)
3) 분석(Enrichment)
키워드/토픽 분류
개체명 인식(후보명, 지역명, 기관명)
유사문서 군집(같은 이슈 묶기)
감성/프레이밍(단, “판정”이 아니라 “신호”로만)
요약 브리프 생성(출처 링크 포함)
World Monitor가 “AI 브리프”로 과부하를 줄인다고 한 부분이 여기입니다. (GitHub)

4) 구현 스택 3가지 옵션: (현실적인 선택지)
옵션 1) “빠른 MVP” (2~4주)
수집: RSS + 일부 공식 API
저장: PostgreSQL + OpenSearch(검색)
대시보드: Next.js(웹) + 지도(Mapbox/Deck.gl)
요약: OpenAI/로컬 LLM(Ollama) 중 택1
알림: Slack/Email 웹훅
→ “지도 + 피드월 + 오늘의 브리핑 + 알림”까지 구현 가능
옵션 2) “World Monitor식”으로 더 닮게 (1~3개월)
레이어가 많은 지도 UX
로컬 RAG(Headline Memory 같은 로컬 인덱스) (GitHub)
API 계약(Proto/OpenAPI)로 팀 분업
후보/이슈 타임라인, 룰 기반 경보 고도화
옵션 3) “오픈소스 World Monitor를 포크해서 개조”
World Monitor는 오픈소스이며(TypeScript 중심), 여러 구성 폴더(api/server/tauri 등)가 있는 형태라 “포크→선거 모듈만 교체”도 가능합니다. (GitHub)
단, 라이선스(AGPL) 의무가 생길 수 있어, 공개 범위/배포 형태를 법무/정책과 같이 판단해야 합니다. (GitHub)

5) “모니터링”의 기준(지표/방법론)을 잡아야 사이트가 설득력을 가집니다
여기서 World Monitor는 “도구”라면, 선거 모니터링은 “방법론”이 핵심이에요. 참고할 만한 국제 가이드는 다음이 가장 정석입니다.
OSCE/ODIHR는 선거를 “하루짜리 이벤트가 아니라 과정”으로 보고 전반 방법론을 제시합니다. (OSCE ODIHR)
**ODIHR ‘미디어 모니터링 핸드북’**은 선거 보도 모니터링 방법(표본추출, 코딩, 균형성 평가 등)을 정리한 자료입니다. (OSCE ODIHR)
NDI는 시민단체 관점의 미디어 모니터링/선거 모니터링 실무 자료를 제공합니다. (국가민주연구소)
**NDI 소셜미디어 모니터링 가이드(2019)**는 선거에서의 소셜 모니터링 접근을 정리합니다. (국가민주연구소)
IFES는 선거 무결성과 정보환경(디스인포메이션 포함)의 위험을 다루는 자료를 제공합니다. (IFES)
➡️ 결론적으로, 사이트에 “우리는 무엇을 어떻게 측정하는가(지표 정의)”를 박아 넣으면 신뢰가 생깁니다.

6) 법·리스크 체크(꼭)
지방선거 모니터링은 아래 리스크를 초기에 정리해야 운영이 지속됩니다.
플랫폼 약관/수집 허용 범위(특히 SNS)
개인정보/민감정보 처리(댓글·사용자 식별정보 수집 금지 또는 비식별화)
허위정보 판정의 책임
“판정”이 아니라 “의심 신호/팩트체크 링크” 중심으로 설계 권장
예: “루머 탐지 → 출처/확산경로 → 팩트체크 연결” 구조
(해외에서도 선거 디스인포 모니터링 대시보드나 플랫폼은 ‘투명성/책임’ 이슈가 크게 다뤄집니다. (Misinformation Dashboard))

7) 바로 실행 가능한 “첫 10일 플랜”(MVP 기준)
D1~2: 목표/범위 결정
모니터링 지역(예: 충북 전체 vs 청주시 중심)
모니터링 대상(시장/군수/도의원/교육감 등)
소스 목록(언론 20, 후보 공식계정, 선관위/지자체 공지)
D3~5: 수집/저장
RSS 수집기 + DB 적재
공통 스키마 확정(지역코드/후보/토픽)
D6~7: 화면 1차
지도(선거구 폴리곤) + 피드 월 + 필터
D8~9: 브리핑/알림
일일 브리핑 자동 생성(출처 링크 포함)
급상승 룰 기반 알림(키워드/증가율)
D10: 운영 리포트 템플릿
“오늘의 이슈 TOP5 / 후보별 리스크 / 내일 주목 포인트” 자동 PDF/메일

다음 단계(딱 2가지만 주면, 내가 “설계도 + 화면 와이어프레임 + 데이터 스키마”까지 구체화할게)
모니터링 대상 선거: (예: 충북 ‘청주시장’ 중심 / 충북 전체 / 특정 시군)
주요 목적: (보도 균형/이슈 추적/허위정보 대응/후보 공약 비교/시민 제보 접수 중 무엇이 1순위?)
이 두 가지만 정해지면, World Monitor 스타일을 그대로 가져와서
화면 구성(메인 1장 + 상세 3장)
데이터 소스 리스트(합법/안정 우선)
스키마/지표 정의
개발 스택(노코드/웹/온프레) 추천
을 “바로 착수 가능한 문서”로 만들어 드릴게요.


1. 충북 전체 2. 이슈 추적

좋아요. **“충북 전체 / 이슈 추적”**이면, World Monitor의 핵심(= 단일 상황판 + 지도 레이어 + 다중 소스 피드 + 요약 브리프 + 알림)을 그대로 가져와서 충북 지방선거 이슈 추적 대시보드로 바꾸면 됩니다.
아래는 **World Monitor를 참고한 구조 분석 → 충북형 기능 설계 → 데이터/모델/운영/리스크까지 포함한 구축 가이드(실행형)**입니다.

0) 목표 정의: “충북 전체 이슈 추적”은 무엇을 성공으로 보나
이슈 추적 사이트의 성공은 “많이 모았다”가 아니라, 이슈의 ‘발생-확산-전환-소멸’을 지역 단위로 읽을 수 있게 하느냐입니다.
발생(Trigger): 어느 지역/어느 채널에서 이슈가 처음 튀었나
확산(Spread): 어떤 매체/커뮤니티/후보 계정이 확산시켰나
전환(Shift): ‘사실 확인/해명/반박/프레임 전환’이 언제 일어났나
소멸(Decay): 언제 관심이 떨어졌나(또는 재점화됐나)
World Monitor도 본질적으로 “흩어진 정보 → 단일 상황판”으로 과부하를 줄인다고 설명합니다. (GitHub)

1) World Monitor 웹사이트(프로젝트)에서 가져올 핵심 구조 5가지
World Monitor는 “실시간 글로벌 인텔리전스 대시보드”로, 뉴스/이벤트를 통합하고(aggregation), 지도와 레이어로 상황을 보여주며, AI 브리프로 요약하는 형태입니다. (GitHub)
이걸 충북 선거 이슈 추적으로 옮기면 아래 5개가 핵심입니다.
1) 단일 상황판(One screen, many signals)
“피드/지도/브리프/알림”을 한 화면에서 순환시키는 구성
‘이슈 추적’에선 **“오늘 충북 TOP 이슈 / 급상승 / 지역별 분포 / 타임라인”**이 메인
2) 지도 + 레이어(Geo-context)
World Monitor는 지리 기반 레이어(다층 레이어)를 핵심 UI로 둡니다. (GitHub)
충북에선 시·군·구(청주/충주/제천/음성/진천/괴산/보은/옥천/영동/단양/증평…) 단위로 “어디에서 이슈가 뜨는지”가 직관적으로 보여야 합니다.
3) AI 브리프(요약)로 과부하 감소
World Monitor는 “AI-powered news aggregation”과 브리프를 전면에 내세웁니다. (GitHub)
선거 이슈 추적에서도 “기사 300건”이 아니라
**“이슈 6개 + 각 이슈의 핵심 주장/근거/반박/출처”**가 가치입니다.
4) RAG(검색/질의응답) = “이슈 히스토리”를 묻고 답한다
World Monitor 릴리즈 노트에 “worker-side vector store, Headline Memory (RAG)” 같은 흐름이 보입니다. (GitHub)
충북형으로 바꾸면:
“지난 2주간 ‘오송역/교통’ 이슈가 어떻게 바뀌었어?”
“청주시에서 ‘개발’ 프레임이 ‘복지’로 바뀐 변곡점이 언제야?”
같은 질문에 출처 붙여 요약해주는 기능이 됩니다.
5) API 계약(깨지지 않는 파이프라인)
World Monitor는 “운영 가능한 인텔리전스”에 가까운 프로젝트라, 수집·정제·서비스 분리를 전제로 가기 쉽습니다(오픈소스 구조 자체가 그런 방향). (GitHub)
선거는 수집원이 자주 바뀌고 차단도 잦아서, 계약 기반 스키마가 없으면 유지보수가 무너집니다.

2) “충북 이슈 추적 대시보드” 화면 설계(= 바로 와이어프레임 가능한 수준)
아래는 가장 실용적인 4화면 구성입니다.
(A) 홈: 충북 상황판(Real-time)
상단 KPI
오늘 신규 이슈 수 / 급상승 이슈 수 / 고위험(논란) 신호 수 / 지역 커버리지(시군구 몇 곳에서 언급 발생)
중앙: 이슈 카드 6~10개
각 카드에 반드시 6요소:
이슈명(자동 생성)
지역(시군구)
상승 속도(전일 대비, 최근 2시간 대비)
주요 채널(언론/유튜브/커뮤니티/후보SNS)
한 줄 브리프(요약)
“근거 링크 3개(대표 출처)”
우측: 브리핑(오늘의 충북 이슈 브리프)
World Monitor식 “brief” 영역을 그대로 이식
‘정리’가 목적이므로 출처 링크를 강제(나중에 논란 방지)
(B) 지도: 지역별 이슈 분포(Map)
시군구 폴리곤 위에 “이슈 열지도(강도)”
레이어 토글:
정책 분야(개발/복지/교육/교통/안전/환경/일자리…)
채널(언론/커뮤니티/후보SNS/공식공지)
시간(최근 2시간/24시간/7일)
(C) 이슈 상세: 타임라인(Trace)
이슈 하나를 클릭하면:
“최초 발견 소스” → “확산 피크” → “반박/해명/팩트체크” → “재점화”
관련 문서 군집(유사 기사/복붙 포스트 묶기)
핵심 인물/기관(개체명)
논점 3개(찬/반/팩트)
(D) 탐색/검색: RAG 기반 “이슈 질의”
키워드 검색 + “질문하기”
답변은 근거 출처(링크) + 요약 + 시간 범위가 세트
(선거 국면에서 가장 강력한 기능이 됩니다)

3) 데이터 소스 설계(충북 전체에 맞게)
이슈 추적은 “정확한 개표/후보 등록” 같은 정형 데이터보다, 비정형 텍스트/영상 메타가 핵심입니다. 소스는 4축으로 나누면 안정적입니다.
1) 언론(지역·전국)
지역 언론, 방송, 통신사 기사 RSS/웹
핵심은 “지역 태깅(충북/청주/충주…)” 정확도
2) 후보/정당 공식 채널
유튜브(채널/영상), 보도자료 페이지, 공식 홈페이지
SNS는 약관/차단 리스크가 커서 “가능한 합법적/공식 제공 범위”부터
3) 지역 커뮤니티/제보
지역 이슈는 커뮤니티에서 먼저 뜹니다.
다만 개인정보/명예훼손 리스크가 크므로 “원문 노출 최소화, 링크 중심, 비식별화” 원칙이 중요
4) 공공/공식 공지
선관위/지자체 공지, 정책자료, 보도해명
(이슈 ‘전환점’에 해당하는 근거가 되기 쉬움)
참고: 한국 선거 데이터는 “웹에서 제공되지만 프로그램 접근이 제한적일 수 있다”는 지적도 있습니다. (Open Data Index)
그래서 “공식 통계/결과”를 핵심 기능으로 잡기보다, 지금 목표(이슈 추적)에 맞게 텍스트/영상 중심으로 설계를 시작하는 게 안전합니다.

4) 이슈 추적 알고리즘(실무형): “문서가 아니라 ‘이슈’를 만든다”
Step 1) 수집 → 공통 스키마 정규화
최소 스키마(권장):
id, source_type(news/community/sns/official)
source_name, url, published_at
title, body, author(가능하면)
region(시군구), geo_hint(주소/지명)
entities(후보/기관/장소)
topic(정책 분야)
engagement(조회/댓글/좋아요가 가능할 때만)
Step 2) 이슈 군집(Clustering)
문서 임베딩(한국어 강한 모델) → 유사도 기반 군집
군집이 “이슈”가 됩니다.
이슈명: 군집 대표 키워드/개체명으로 자동 생성
Step 3) 이슈 스코어링(급상승/중요도)
볼륨(문서 수) + 속도(증가율) + 분산(여러 소스에서 동시에)
지도 스코어: 지역별 언급 비율
Step 4) 타임라인/변곡점 감지
“반박/해명/사과/팩트체크/고소/수사” 같은 전환 단어
소스 다양화(언론 → 공식 → 커뮤니티) 흐름 변화
이게 “이슈 추적”의 핵심 가치
Step 5) 브리프 생성(출처 강제)
World Monitor가 ‘AI 기반 상황판’을 표방하는 것처럼, 브리프는 필수입니다. (GitHub)
단, 선거/정치 영역은 민감하니 “판정형 문장”은 피하고
“A 주장 / B 반박 / 확인된 사실 / 아직 불명” 구조로 작성

5) 운영 방법론(신뢰/투명성): 국제 가이드에서 가져올 ‘틀’
이슈 추적이 정치적으로 민감해질수록, “우리 사이트가 뭘 어떻게 보고 있나”를 공개해야 공격을 덜 받습니다.
여기서 OSCE/ODIHR 자료가 실무 틀을 제공합니다.
ODIHR 미디어 모니터링 핸드북은 선거 기간 미디어 모니터링의 목적/방법(표본, 코딩, 균형성 등)을 정리합니다. (OSCE)
NDI(2019) 소셜미디어 모니터링 가이드는 선거 시기 소셜 모니터링의 기법/주의점을 다룹니다. (국가민주연구소)
Open Society 자료도 선거 무결성과 소셜 미디어 모니터링 이슈를 다룹니다. (오픈소사이어티재단)
UNESCO의 “디지털 시대 선거 가이드”도 참고 프레임으로 좋습니다. (유네스코 문서)
충북형 운영 원칙(권장 7개)
출처 링크 우선(원문/공식/신뢰매체)

이 카드만 잘 만들어도, 회의는 사실상 끝납니다. “오늘 뭐하지?”가 사라져요.
A-4. 우측 패널: “오늘 회의용 브리프”
자동 생성 문서(회의록에 그대로 붙여넣기 좋게)
(1) 오늘 충북 TOP 이슈 5개(각 이슈 3줄 요약 + 링크 3개)

(5) 내일 주목 포인트(예약/일정/행사 등 붙일 수 있으면)

(화면 B) 지도: “충북 이슈 히트맵 & 확산 경로”
목표: “어디에서 불이 났고, 어디로 옮겨갔는지”를 20초에 확인
B-1. 기본 맵
충북 시군구 폴리곤
색 강도 = 이슈 강도(가중합)
B-2. 레이어 토글
토픽 레이어(개발/교육/복지/논란…)
채널 레이어(SNS/커뮤니티/언론)
대상 레이어(시장군수/교육감)
시간 레이어(2시간/24시간/7일)
“확산 화살표”(선택):
SNS에서 시작 → 커뮤니티 → 언론으로 넘어가는 흐름을 선으로 표시
또는 지역 간 전이(청주 → 인근 시군으로 확산) 표시
B-3. 지도 클릭 시
해당 지역 TOP 이슈 5개
“지역 키워드 클라우드”
“지역 후보 관련 언급(시장군수/교육감) 요약”

(화면 C) 이슈 상세: “Trace 타임라인”
목표: 이슈 하나를 “기사 모음”이 아니라 “스토리의 흐름”으로 이해
C-1. 이슈 개요
이슈명 / 시작 시각 / 피크 시각 / 현재 상태(상승/정체/하락/재점화)
지역 / 대상(시장군수/교육감)
채널 구성(도넛/바) + 소스 다양성 지표(언론만? 커뮤니티만?)
C-2. 타임라인(핵심)
시간순 이벤트로 정리:
최초 발견(첫 등장 링크)
확산(공유/리포스트/커뮤니티 게시 급증)
언론 반영(기사화)
전환(반박/해명/정정/팩트체크/기관 공지)
재점화(다른 이슈와 결합, 2차 논란)
C-3. “논점 카드” 3개(찬/반/팩트 형태가 아니라 ‘주장/반박/확인필요’)
주장 A 요약(출처 링크)
반박/해명 요약(출처 링크)
확인 필요 포인트(취재 질문)
C-4. 유사 문서 군집
같은 내용 반복(복붙 기사/밈/짤)을 묶어서 “노이즈 제거”
운영자가 “이건 같은 이슈/다른 이슈”를 수정할 수 있어야 함(매우 중요)

(화면 D) 탐색/검색: “이슈 지식창고(RAG)”
목표: 회의 중에 데스크가 묻는 질문에 “근거 딸린 답”을 10초~30초에 제공
예:
“최근 7일 충북에서 교육감 관련 논란은 뭐가 컸지?”
“청주 개발 이슈가 왜 다시 뜬 거야? 변곡점 링크 줘.”
“이 이슈는 어디에서 시작했어? SNS 원발 찾을 수 있어?”
D-1. 검색 모드 2가지
키워드 검색(정확)
질문하기(자연어) → 답변 + 근거 링크 5개
D-2. 답변의 형식 고정(편집회의용)
요약(5줄 이내)
타임라인 핵심 3개
근거 링크(최소 5개)
“주의: 확인되지 않은 주장 포함 가능” 고지 자동 삽입

3) 데이터 구조(스키마): “문서 → 신호 → 이슈” 3층으로 설계
이슈 추적은 “기사 DB”가 아니라, 신호와 이슈가 별도로 있어야 합니다.

3-1. RawDoc (원문 단위)
doc_id (PK)
source_type : sns | community | news | official
platform : youtube | instagram | facebook | x | blog | cafe | dc | news_site ...
source_name : 계정/커뮤니티/언론사 이름
url
published_at
fetched_at
title
body_text (가능하면)
media_type : text | video | image | mixed
media_meta : 길이, 썸네일, 해시태그 등
author_handle (공개된 경우만, 내부 표시 최소화 권장)
engagement : views/likes/comments/shares (가능한 항목만)
region_hint : 본문에서 추출된 지명/주소 힌트
candidate_hint : 후보/정당/교육감 관련 이름 힌트
topic_hint : 토픽 분류 결과(초안)
risk_flags : 개인정보/혐오/명예훼손/불법촬영 등 신호
hash_sim : 유사도/중복 제거를 위한 해시(예: SimHash)
SNS 50% 비중을 충족하려면, RawDoc에서 engagement와 platform이 필수입니다.
“조회수 급증 + 공유 급증”이 편집회의에서 가장 강한 신호거든요.

3-2. Signal (시간창 기반 지표)
RawDoc를 그대로 보여주면 노이즈가 많습니다. 그래서 “신호”를 만듭니다.
signal_id
time_window : 10분/30분/2시간/24시간
region : 시군구
race_type : mayor_county | education_superintendent
topic
doc_count_total
doc_count_sns, doc_count_comm, doc_count_news
engagement_score : SNS 가중치 반영
growth_rate : 이전 창 대비 증가율
source_diversity : 서로 다른 소스 수(계정/커뮤니티/언론)
risk_score : 위험 플래그 누적

3-3. IssueCluster (이슈 단위)
이게 화면 A의 “이슈 카드”가 됩니다.
issue_id

region (기본) + region_spread (확산된 지역 리스트)
race_type
topic

risk_flags / risk_score
manual_override : 운영자 수정 이력(감사/투명성)

4) 이슈 스코어(편집우선순위) 설계 — “SNS/커뮤니티가 먼저 뜨게”
사용자 요구가 SNS 50 / 커뮤니티 30 / 언론 20이므로, 점수식 자체에 그 비중을 박아 넣습니다.

4-1. 기본 점수 구성(권장)
IssueScore = 0.40*Velocity + 0.25*Reach + 0.20*Diversity + 0.10*Impact + 0.05*EditorialFit - Penalty
(1) Velocity(상승속도) — 가장 중요
최근 2시간 문서 증가율 + 최근 30분 증가율 혼합
특히 SNS/커뮤니티는 “짧은 시간 급증”이 강력
(2) Reach(도달/반응)
SNS의 engagement 기반
예: 조회수/좋아요/댓글/공유를 로그 스케일로 합산
(3) Diversity(소스 다양성)
서로 다른 계정/커뮤니티/언론이 동시에 언급하면 신뢰도/의제화 가능성↑
“한 계정/한 커뮤니티에서만” 돌면 조작/왜곡 가능성이 높아 Penalty 쪽으로 반영
(4) Impact(영향)
선거 관련성(후보/교육감 직접 연관)
공공성 높은 분야(안전/교육/부패/선거법 논란) 가중
(5) EditorialFit(편집 적합도)
내부 편집회의용이므로 “기사화 가능성” 신호
언론 반영(20%)이 낮더라도, 언론이 붙기 시작하면 Fit이 올라감
즉, 언론은 ‘초기 발견’보다 ‘의제화 확정’ 신호로만 쓰기 좋습니다.
Penalty(감점)
개인정보 노출/혐오/명예훼손 위험
출처 불명/캡처본만 존재/원발 추적 불가
동일 문구 반복(봇/조직적 확산 의심)
선거법/플랫폼 약관 위반 소지 콘텐츠 노출 위험

4-2. 채널 가중치(요구조건을 스코어에 반영)
ChannelWeight = 0.5*SNS + 0.3*Community + 0.2*News
SNS 신호가 먼저 상단을 차지하게 됨
다만 “언론이 붙는 순간” Fit을 올려서, 회의에서 “아이템화”로 가는 브릿지를 만듭니다.

5) 이슈 생성/병합 로직 — 운영에서 무조건 부딪히는 부분
실무에서 가장 많이 터지는 게 이슈가 쪼개지거나 뭉개지는 문제예요.
5-1. 자동 클러스터링 기본 규칙(권장)
1차: 키워드/개체명(지역+후보명) 기반 후보군 생성
2차: 임베딩 유사도 기반 군집
3차: 시간창(예: 24시간) 안에서만 군집 우선
4차: 운영자가 “병합/분리”를 클릭하면, 그 선택이 학습(룰)로 남도록
5-2. 병합/분리 운영 규칙(편집회의용)
병합 기준: “회의에서 같은 이슈로 취급할 수 있나?”
분리 기준: “대상(시장군수/교육감)이 다르거나, 지역이 다르거나, 논점이 다르면 분리”
운영자 UI에 “이 이슈와 유사한 이슈 후보 5개”를 자동 추천하면, 운영 부담이 급감합니다.

6) 알림(Alerts) — 편집회의에 실제로 도움이 되는 룰만
알림이 많으면 사람들은 끕니다. 내부 편집회의용은 짧고 강한 룰 6개면 충분합니다.

6-1. 알림 룰 6종(권장)
급상승 알림
최근 30분 대비 2시간 증가율이 임계치 초과(예: 3배)
채널: SNS 또는 커뮤니티에서 먼저 발생하면 우선 알림
지역 전이 알림
한 지역에서 시작한 이슈가 2개 이상 시군구로 확산
언론 진입 알림(아이템화 신호)
SNS/커뮤니티 이슈가 언론 기사로 첫 등장했을 때
내부 회의에서 “이제 방송/기사로 다룰 타이밍” 판단 근거
교육감 전용 경보
교육감 이슈는 전체 이슈에서 비중이 작아 묻히기 쉬움 → 별도 룰
리스크 경보(법무/윤리)
개인정보/명예훼손 플래그가 일정 점수 이상
“노출 금지/원문 링크 제한/내부만 공유” 같은 조치 자동 제안
원발 부재 경보(조작 위험)
캡처/2차 유통만 많고 원문 URL이 없을 때
“확산은 큰데 근거가 약함”으로 표시 + 알림

6-2. 알림 메시지 포맷(회의용 6줄 고정)
[급상승] 이슈명 / 지역 / 대상(교육감/시장군수)
2시간 증가율: ▲x.x배
채널 구성: SNS xx% / 커뮤니티 xx% / 언론 xx%
핵심 키워드 3개
대표 링크 3개
편집 제안: “추적/팩트확인/아이템화”

7) 내부 편집회의 “사용 시나리오”(이게 있어야 정착합니다)
사이트가 좋아도, 회의 루틴에 못 붙으면 안 씁니다. 그래서 사용 루틴을 “의식적으로” 설계해야 합니다.
회의 15분 루틴(권장)
(2분) 홈 화면 A: TOP 이슈 카드 10개 훑기
(3분) 지도 B: 지역 분포 확인 → “오늘 지역 균형” 체크
(7분) 이슈 상세 C: TOP 3만 타임라인 확인 + 취재 질문 뽑기
(3분) 브리프 복사: “오늘 회의 결론”을 내부 공유 채널에 게시
회의 산출물(자동 생성)
“오늘의 TOP 5 이슈”
“오늘의 취재 질문 10개”
“위험 신호 이슈 3개(주의 문구 포함)”
“교육감 이슈 3개(별도 섹션)”
이걸 매일 쌓으면, 자연스럽게 “충북 선거 이슈 아카이브”가 생깁니다.

8) 리스크/법·윤리 안전장치(선거 모니터링은 여기서 승부 납니다)
정치/선거 모니터링은 잘못 설계하면 명예훼손/허위사실 유포/개인정보 이슈에 바로 걸립니다. 그래서 애초에 UI와 문장을 “안전하게” 고정합니다.
8-1. UI 안전장치
커뮤니티 글은 원문 전문 노출 최소화(요약 + 링크)
인물 실명/전화번호/주소 등이 감지되면 자동 마스킹
“확인되지 않은 주장” 배지 자동 표시
운영자만 볼 수 있는 내부 코멘트 기능(외부 공개 금지)
8-2. 문장 안전장치(요약 생성 규칙)
금지: “~이다(단정)”
권장: “~라는 주장이 확산”, “~에 대한 반박/해명 존재”, “확인 필요”
모든 요약에는 근거 링크 포함(최소 3개)
8-3. 운영 프로세스(정정/삭제 요청 대비)
(1) 접수 → (2) 링크 확인 → (3) 카드 내 수정(요약/태그) → (4) 이력 남김
이력을 남겨야 내부 감사/분쟁 대응이 됩니다.

9) 구현 스택 제안(현실형) — “MBC 내부 환경”도 고려한 2안
여기서는 기술을 ‘멋있게’보다 ‘안정적으로’ 선택하는 게 중요합니다.
A안: 빠른 MVP(3~5주) — 추천
웹: Next.js(대시보드)
DB: PostgreSQL
검색/유사도: pgvector(간단) 또는 OpenSearch(규모 커지면)
백엔드: Node/Python 어느 쪽도 가능(팀 역량 기준)
배포: 내부망 가능(보안 필요 시), 아니면 Vercel+보안 게이트
요약:
내부 민감도 높으면 로컬 LLM(Ollama 등)
품질 우선이면 클라우드 LLM + 데이터 최소 전송 전략

1. 배경 & 문제 정의
1.1 배경
지방선거 국면에서 충북 전역(11개 시·군 + 청주시 구 단위 포함)은 이슈가 SNS→커뮤니티→언론 순으로 빠르게 이동한다. 내부 편집회의에서 필요한 것은 “기사 목록”이 아니라, **오늘의 의제(agenda)**를 선점/추적하고 취재·편집 결정을 빠르게 내릴 수 있는 인텔리전스 UI다.
1.2 현재 문제(As-Is)
SNS/커뮤니티에서 먼저 터진 이슈가 회의 시간 전에 이미 확산되는데, 내부에 “한 화면”으로 모인 상태가 없다.
언론 기사 중심 모니터링은 늦고, 커뮤니티/숏폼 기반 논쟁은 발견이 느리다.
회의 때마다 “오늘 뭐 잡지?” 판단이 개인 감에 의존한다.
위험한 이슈(명예훼손/허위·조작/개인정보)도 빠르게 확산하지만, 조기 경보 체계가 약하다.
1.3 목표(Goal)
충북 전역에서 발생하는 선거 관련 이슈를 실시간으로 탐지/군집화하여 “이슈 카드”로 제공
SNS/커뮤니티 중심(50/30/20)으로 급상승/확산/전환점을 보여줌
편집회의에서 2~5분 내 오늘 다룰 이슈를 선정하고 “취재 질문”을 도출
위험 신호(명예훼손/개인정보/조작 의심)를 ‘판정’이 아니라 ‘신호’로 조기 표시
1.4 비목표(Non-Goal)
선거 결과(개표) 집계/공식 통계 서비스가 핵심이 아님
특정 후보/정당 유불리 분석(평가/선호) 기능 제공하지 않음
허위정보 “판정”을 자동으로 확정하지 않음(팩트체크 ‘연결’ 또는 ‘확인 필요’ 수준)

6. 이슈 스코어링(편집우선순위 점수) — 기본값 제안
요구 채널 비중 반영: SNS 0.5 / 커뮤니티 0.3 / 언론 0.2
6.1 구성요소(기본)
IssueScore =
0.40 * Velocity
0.25 * Reach
0.20 * Diversity
0.10 * Impact
0.05 * EditorialFit
Penalty
Velocity(상승속도)
v1 = doc_count(2h) / max(doc_count(prev_2h), 1)
v2 = doc_count(30m) / max(doc_count(prev_30m), 1)
Velocity = 0.7log(v1) + 0.3log(v2)
Reach(도달/반응)
SNS engagement 기반(가능한 지표만)
Reach = log(1 + viewsα + likesβ + commentsγ + sharesδ)
α/β/γ/δ는 플랫폼별로 초기 값 설정 후 보정
Diversity(소스 다양성)
서로 다른 source_name 수 + 서로 다른 platform 수
Diversity = log(1 + unique_sources + 0.5*unique_platforms)
Impact(영향)
후보/교육감 명시 + 토픽(교육/선거법/청렴 등) 가중
내부 규칙표로 관리(운영자 수정 가능)
EditorialFit(아이템화 가능성)
언론 진입(첫 기사 등장) + 공식 해명/공지 등장 시 가중
Fit = 1(언론 진입) + 0.5(공식 문서) 정도로 단순하게 시작
Penalty(감점)
PII/명예훼손/혐오/불법 의심 플래그 점수화
원발 부재/캡처 편중, 동일 문구 반복(봇 의심) 감점
6.2 채널 믹스 반영
각 구성요소 계산 시 doc_count를 채널별로 가중합:
WeightedDocCount = 0.5SNS + 0.3Comm + 0.2*News
이 값을 Velocity 계산에 사용하면 “SNS 중심” 정렬이 자연스럽게 구현됨.

7. 이슈 생성/군집(Clustering) — 운영 가능성 중심 설계
7.1 기본 파이프라인(Phase 1)

대표 링크 선정(각 채널 상위 1~2개씩)

좋아. 내가 아까 말한 “Node로 갈지 Python으로 갈지”, “지도는 Mapbox인지 대체인지”는 개발 방식(팀 구성·운영·비용·성능)이 통째로 달라지는 분기점이라서, 여기서 한 번만 잘 정하면 이후가 엄청 편해져요.
지금 너희 요구(충북 전체 / 이슈 추적 / SNS·커뮤니티 중심 / 내부 편집회의용)에 맞춰 선택 기준 + 추천 조합 + 실제 시스템 구조까지 아주 구체적으로 설명할게.

1) Node vs Python: 무엇이 달라지나
둘 다 “서버”를 만들 수 있지만, 이 프로젝트에선 서버가 하나가 아니라 보통 3개 역할로 나뉩니다.
웹 API 서버: 대시보드가 호출하는 API(이슈 리스트, 지도 데이터, 상세, 검색 등)
수집/배치 워커: RSS/채널/페이지를 주기적으로 수집하고 DB에 적재
AI/분석 워커: 임베딩, 군집, 요약 브리프 생성, 이상탐지
Node/Python 선택은 “3개 중 뭘 어디서 맡길지” 문제예요.

2) Node를 선택하면 좋은 경우
장점
대시보드/API 개발 속도가 빠름(Next.js랑 궁합 좋음)
실시간 기능(웹소켓, SSE) 붙이기 쉬움
프론트/백을 JS/TS로 통일 → 인력 운용이 편함
타입스크립트로 스키마(계약) 관리가 깔끔함(특히 OpenAPI/프론트 연동)
단점(이 프로젝트에서 실제로 부딪히는 포인트)
텍스트 분석/ML/임베딩/클러스터링은 Python 생태계가 압도적으로 편합니다.
“군집/임베딩/NER/토픽”을 Node에서 끝까지 하려면, 결국 외부 서비스 호출이나 별도 구현 부담이 커져요.
Node가 특히 좋은 파트
API 서버(권장)
실시간 알림/웹훅(슬랙 알림)
관리자 콘솔/권한/감사로그
프론트와 타입 공유

3) Python을 선택하면 좋은 경우
장점
이 프로젝트의 핵심인 이슈 군집/임베딩/RAG/요약을 만들기 가장 쉬움
(문서 벡터화, HDBSCAN/DBSCAN, 토픽 분류, NER 등)
데이터 파이프라인(배치/ETL) 구축 경험이 있는 인력이 많음
모델 튜닝/실험(오탐·누락 줄이기)이 빠름
단점
웹 API/권한/실시간 업데이트 같은 “웹앱 운영” 부분은 Node만큼 빠르진 않을 수 있음
(물론 FastAPI로 충분히 가능하지만, 조직에서 프론트 개발자와 협업이 JS 중심이면 통합이 덜 매끄럽기도 함)
Python이 특히 좋은 파트
수집/배치 워커(권장)
AI 분석 워커(강추)
임베딩/군집/요약/이상탐지 파이프라인
데이터 품질 점검/QC

4) 결론: 이 프로젝트에 제일 안전한 “실무형 추천 조합”
너희 케이스(내부 회의용, 빠른 MVP, SNS/커뮤니티 비중 큼)에서 가장 실패 확률 낮은 조합은:
✅ 추천 1안(가장 현실적)
API/대시보드 = Node(TypeScript)
수집·AI 분석 = Python
왜냐면,
화면/권한/필터/UI는 Node+Next가 빠르고
이슈의 품질(군집/요약/리스크)은 Python이 압도적으로 유리하기 때문입니다.
✅ 추천 2안(팀이 Python 강하고 웹은 단순하게 갈 때)
API도 Python(FastAPI)
프론트는 Next.js 그대로 쓰되, API만 FastAPI로 제공
분석/수집도 Python으로 통일
이건 “웹 운영 경험이 Python에 충분히 있을 때” 좋습니다.
⚠️ 비추천(초기에 흔히 빠지는 함정)
Node로 전부 해결(분석까지 JS로 억지로)
→ 시간이 갈수록 “군집/요약 품질 개선”에서 발목 잡힙니다.

5) “지도는 Mapbox인지 대체인지” — 왜 중요한가
지도는 그냥 배경이 아니라, 이 서비스에서 핵심 UX예요.
충북 이슈 추적은 “어디에서 이슈가 뜨는지”가 가치이므로 지도 품질이 사용성에 직결됩니다.
지도 선택에서 현실적으로 고려해야 할 건 4가지:
벡터 타일/성능: 시군구 폴리곤을 자주 칠하고(heat), 레이어 토글이 많음
비용/라이선스: 내부용이라도 접속량이 늘면 과금 리스크
구현 난이도: 개발팀이 지도 경험 없으면 개발 기간 폭발
배포 환경: 사내망/폐쇄망이면 외부 타일 서비스 사용이 제한될 수 있음

6) Mapbox를 쓰면 좋은 경우
장점
지도 UX(줌/레이어/성능)가 안정적이고 예제가 많음
Deck.gl 같은 시각화 라이브러리랑 조합이 좋음(폴리곤/히트맵/아크 라인)
개발 시간이 확 줄어듦(특히 MVP)
단점
비용/약관 고려 필요(트래픽 증가 시)
사내망/폐쇄망에서 외부 호출이 막히면 곤란
Mapbox 추천 상황
MVP를 빠르게, 지도 퀄리티를 안정적으로 내야 할 때
외부 인터넷 사용이 가능한 환경(또는 예외 허용)

7) “대체(오픈) 지도” 옵션은 무엇이 있나
여기서 “대체”는 크게 2가지로 나뉩니다.
(A) MapLibre GL (Mapbox의 오픈소스 계열)
Mapbox처럼 벡터 지도 스타일/상호작용을 만들 수 있음
타일은 OpenStreetMap 기반 벡터 타일(또는 자체 호스팅 타일) 사용 가능
장점: 비용 유리, 자체 운영 가능, Mapbox와 코드 유사
단점: 타일/스타일 운영을 직접 챙겨야 할 수도 있음(=DevOps 부담)
(B) Leaflet
가볍고 간단한 지도
하지만 벡터 타일/복잡 레이어 성능은 Mapbox/MapLibre보다 약함
장점: 배우기 쉬움, 단순 표시엔 충분
단점: “레이어가 많은 상황판”은 점점 무거워질 수 있음

8) 지도 선택: 너희 조건(내부 편집회의용, 충북 전체)에 딱 맞춘 추천
✅ 1순위(대부분 팀에 최적)
Mapbox + Deck.gl
이유: MVP에서 “지도 레이어/히트맵/확산 화살표”를 가장 빨리, 가장 안정적으로 구현
✅ 2순위(비용/폐쇄망이 걸리면)
MapLibre + (가능하면) 자체 타일 또는 안정적 타일 제공원
이유: Mapbox와 비슷한 개발 경험 + 비용/통제 이점
단순히 ‘지역별 리스트’만 보여줄 거면
Leaflet도 가능하지만, 너희는 “확산 흐름/레이어 토글”이 핵심이라 1~2순위가 훨씬 맞아요.

9) 그래서 “선택을 알려주면” 내가 뭘 해줄 수 있냐(구체 산출물)
네가 “Node로 갈게 / Python으로 갈게 / 지도는 Mapbox” 같은 결정을 주면, 나는 바로 아래처럼 개발 문서가 한 단계 더 구체화됩니다.
9.1 API 계약(OpenAPI) 설계가 확정됨
예를 들어 Node(TypeScript)로 API 서버를 간다고 하면,
/api/issues (이슈 리스트, 필터/정렬)
/api/issues/{id} (이슈 상세/타임라인)
/api/map/regions (시군구 폴리곤/속성)
/api/map/heat (시간창/토픽별 강도)
/api/brief/today (회의용 브리프)
/api/alerts (알림 룰/로그)
/api/admin/* (병합/분리/숨김/블랙리스트)
이걸 타입까지 맞춰서 프론트와 “깨지지 않게” 붙일 수 있어요.
9.2 워커 파이프라인 스케줄도 확정됨
Python 워커 기준이면,
5~15분마다 수집
30분마다 임베딩/군집 업데이트
10분마다 급상승/경보 계산
매일 오전 8:30 브리프 생성(회의 전)
이런 배치 구성이 자연스럽게 떨어집니다.
9.3 지도 구현 방식이 확정됨
Mapbox면
시군구 폴리곤은 GeoJSON으로 제공
Heat는 지역별 intensity 숫자만 내려줘도 됨
확산 화살표는 “source region → target region” edge list로 내려주면 됨
MapLibre/자체 타일이면
타일/스타일 호스팅까지 포함한 운영 문서가 필요합니다(이건 개발 범위가 커져요).

10) 지금 상황에서 내가 “best effort”로 추천 결론(확정 제안)
너희 목적(내부 편집회의용) + 속도(MVP) + 분석 난이도(이슈 군집/요약)가 핵심인 걸 고려하면:
API/웹 = Node(TypeScript, Next.js)
수집/AI/군집 = Python(FastAPI는 선택, 워커는 필수)
지도 = Mapbox + Deck.gl (MVP 우선)
비용/폐쇄망 이슈가 나오면 MapLibre로 전환하는 로드맵을 Phase 2에 둠
이 조합이 “빨리 만들고, 이후 품질 개선도 쉬운” 정석입니다.

원하면 다음 메시지에서, 위 추천 조합을 전제로 **“구체 API 목록(OpenAPI 목차) + 배치 스케줄 + DB 인덱스 + 알림 웹훅 포맷 + 지도 데이터 포맷(GeoJSON/heat/edge)”**까지 한 번에 내려줄게.



좋아. 지금부터는 내가 아까 추천한 조합을 확정안으로 두고(= 웹/API: Node(Typescript/Next) + 분석/수집: Python 워커 + 지도: Mapbox+Deck.gl), 개발팀/외주가 바로 착수할 수 있는 수준으로 **구현 명세(Endpoints / 데이터 포맷 / 배치 스케줄 / 알림 웹훅 / 인덱스 설계 / 권한)**까지 한 번에 내려줄게.
